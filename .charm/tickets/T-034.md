Questions to answer concretely:

1. RECEIVING CHAT MESSAGES (most up-to-date path). What is the current, recommended way to receive chat messages programmatically as of 2026? Cover: (a) the legacy IRC/TMI interface (irc.chat.twitch.tv) and its current support status, vs (b) EventSub (channel.chat.message) over WebSocket or webhook transport. Which is now recommended/required, what scopes and token type each needs, subscription cost/rate limits, and what a minimal message-receive flow looks like. Note any announced deprecation timeline for IRC.

2. WHISPERING A USER. Can a bot/app send a whisper (DM) to a user, e.g. in response to a command they typed? Cover the Helix 'Send Whisper' endpoint: exact endpoint, required scope (user:manage:whispers?), token type (user access token for the SENDING account), and the hard prerequisites/gotchas -- notably the verified-phone-number requirement on the sending account, the rule about only whispering users who you have a relationship with / haven't blocked, the 'whisper after the user messages you' constraints if any, and the rate limits (per-second / per-minute / per-day, and the unique-recipient limits that throttle unsolicited whispers). Be explicit about whether unsolicited 'reply to a command with a whisper' is actually viable at scale or whether Twitch's anti-spam rules effectively block it.

3. ALTERNATIVES if whisper is impractical: posting an @mention reply in-channel, the Helix Send Chat Message endpoint, or chat replies (reply-parent). Briefly compare.

4. PRACTICAL VERDICT for our use case (a Twitch chat -> on-screen character playground): given the constraints, recommend the receive mechanism and whether to whisper or reply in-channel, with the rationale.
