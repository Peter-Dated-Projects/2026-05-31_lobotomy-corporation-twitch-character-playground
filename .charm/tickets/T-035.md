RESEARCH ONLY -- no code changes. Write findings to .charm/kb/twitch-receive-and-whisper.md (create it). This feeds a design decision: use whispers (DMs) as a two-way private command channel for the on-screen character playground -- users send commands to the bot via whisper, and the bot can optionally whisper back -- so commands don't clutter public chat.

Use WebSearch/WebFetch against official Twitch developer docs (dev.twitch.tv). Prefer primary sources over blogs, and flag the recency of anything that changed -- Twitch has deprecated several older paths, so 'current as of 2026' matters.

Answer concretely:

1. RECEIVING PUBLIC CHAT MESSAGES (current path). Legacy IRC/TMI (irc.chat.twitch.tv) and its support status vs EventSub channel.chat.message over WebSocket/webhook. Which is now recommended/required, scopes + token type each needs, subscription/rate-limit cost, minimal receive flow, and any IRC deprecation timeline.

2. RECEIVING WHISPERS (user -> bot). THIS IS CENTRAL. How does a bot programmatically receive whispers sent TO it as of 2026? Cover: (a) legacy IRC WHISPER messages and current status, and (b) EventSub user.whisper.message (or whatever the current type is) -- transport, exact scope (user:read:whispers / whisper:read?), token type (user access token for the bot account), and rate/subscription limits. Confirm whether unsolicited inbound whispers from arbitrary viewers are actually deliverable to the bot or whether anti-spam gating blocks them.

3. SENDING WHISPERS (bot -> user). Helix 'Send Whisper' endpoint: exact endpoint, scope (user:manage:whispers?), token type (sending account's user token), and the HARD prerequisites/gotchas -- the verified-phone-number requirement on the sending account, restrictions on whispering users with no prior relationship / who blocked you, and rate limits (per-sec / per-min / per-day plus the unique-recipient cap that throttles unsolicited whispers). Be explicit: is replying to a command with a whisper viable at scale, or do anti-spam rules effectively block it?

4. END-TO-END VIABILITY of the 'commands via whisper' design. Tie 2+3 together: can a viewer realistically whisper a command to the bot and get a whisper reply, accounting for phone-verification, relationship/anti-spam gating, and rate caps? Note the per-account setup burden (the bot account needs a verified phone, etc.).

5. ALTERNATIVES if whisper-as-command-channel is impractical: in-channel @mention replies, Helix Send Chat Message, chat replies (reply-parent), or a channel-point/redeem path. Brief comparison.

6. PRACTICAL VERDICT for our use case. Recommend the receive mechanism (chat and/or whisper) and whether the bidirectional-whisper command channel is workable or we should fall back to in-channel commands, with rationale.

Structure the kb doc with a header per section, cite doc URLs inline, end with a 'Recommendation' section.
