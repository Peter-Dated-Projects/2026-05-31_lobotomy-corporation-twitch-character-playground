Wire the three speak-feature pieces (TTS engine T-004, swear filter T-005-no: filter is T-007, render T-005) together into the running app: a viewer triggers a speak, the message is filtered, synthesized in a roster voice, played, and the robot body + speech balloon show on screen for the duration.

IMPORTANT dep IDs: TTS engine = T-004, render = T-005, swear filter = T-007. This ticket depends on all three.

## Architecture recap (read these files first)
- `twitch_playground/main.py`: owns the loop; drains a `queue.Queue[ChatCommand]`, calls `world.handle_command(cmd)`, then `world.update`, `scene.draw`, HUD/notice draw, then scales the stage. The notice mechanism (transient on-screen text with a monotonic expiry) is a good pattern to mirror for the speak overlay lifecycle.
- `twitch_playground/chat/commands.py`: `ChatCommand(cmd, args, author)`, `parse_command` (only `!`-prefixed). Add the speak command here.
- `twitch_playground/sim/world.py`: `World.handle_command` dispatch table. Add the speak handler + the active-speaker state here.
- `twitch_playground/settings.py`: add all speak-feature config here (this ticket OWNS settings.py for the feature).

## What to build
1. **Command**: support `!say <message>` (v0 trigger; real Twitch channel-point redemptions need EventSub + new scopes -- out of scope, but leave a clear seam / comment for adding a redemption source that enqueues the same internal speak request). parse_command already lowercases cmd and splits args; the message is `' '.join(cmd.args)`.
2. **Character/voice pairing**: deterministic md5(username) -> one of the roster (hod, malkuth, netzach, yesod) so a viewer always speaks through the same Sephirah. Put the roster + hashing in settings.py / world.
3. **Pipeline**: on `!say`: `filter_message(text)` (from speak.filter) -> `SpeakEngine.speak(filtered, character)` (from speak.tts, background thread) -> set active-speaker state {character, text, started}. When the engine signals playback ended (its is-speaking flag/callback), clear the state. Guard against overlap (queue or ignore a new !say while one is speaking -- choose the simpler: ignore-if-busy for v0, documented).
4. **Render hook**: in main.py, after scene.draw, if there is an active speaker call `render.speech.draw_speech(stage, character=..., text=..., font=...)`. Reuse a pygame font (e.g. the world nameplate font or a new one).
5. **Engine lifecycle**: construct `SpeakEngine` once at startup in main.py (it is slow to init). If construction fails (no torch/models in this env) or the feature is disabled via settings/env, degrade gracefully -- the game runs without the speak feature, no crash. Log a clear line.
6. **settings.py**: `SPEAK_ENABLED` (default read from env), `ROBOT_ROSTER = ['hod','malkuth','netzach','yesod']`, voices dir path, font size, overlay anchor, max message length, etc.

## Constraints
- Touch ONLY: main.py, sim/world.py, chat/commands.py, settings.py. Do NOT edit speak/*, render/speech.py, scene.py, provider.py (other tickets own those; consume their public APIs).
- The sim loop must never block on synthesis/playback -- all of that is on the engine's background thread (T-004 provides it).
- Keep dev-mode usable: the DevInjector (twitch_playground/dev.py) can be left alone, but a `!say` typed via the existing command path should work.

## Definition of done
- Typing/sending `!say hello chat` enqueues a speak: message is filtered, the assigned robot + balloon render, audio plays on its own channel, overlay clears when audio ends.
- With SPEAK disabled or engine init failing, the app still runs normally.
- No blocking of the render loop during synthesis.
