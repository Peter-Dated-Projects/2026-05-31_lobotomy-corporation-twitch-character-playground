LobCorpProvider.get_sprite_set caches one SpriteSet per character_id (the viewer username, since T-018). With the L5 emotion variants each SpriteSet is ~240KB (3 emotions x 16 frames x 32x40), so memory scales with UNIQUE VIEWERS even though there are only 10 distinct character looks — 1000 viewers ~= 240MB of cache for 10 real looks. Fix: cache by the RESOLVED character (the CharacterDef name from _resolve), so the cache has at most ~10 entries regardless of viewer count, keeping memory flat.

IMPLEMENTATION: in get_sprite_set, resolve character_id -> CharacterDef name first, then cache/lookup the built SpriteSet by that resolved name. Two usernames that map to the same character share one SpriteSet object (fine — SpriteSets are immutable frame lists; per-character runtime state like emotion/anim lives on Character, not the SpriteSet). Preserve the graceful placeholder fallback (a character that fails to composite still falls back, keyed appropriately). Keep behavior identical for callers.

VERIFY: add/extend a test asserting two distinct usernames resolving to the same CharacterDef get the SAME cached SpriteSet object, and that cache size stays bounded by the number of distinct resolved characters across many usernames. Full suite green. Touch only the listed files.
