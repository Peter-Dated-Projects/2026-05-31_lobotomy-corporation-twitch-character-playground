The per-frame neighbor interaction is O(N^2) (every character vs every other), which is the wall on crowd size: benchmarked ~7ms at N=100, ~28ms at N=200, ~112ms at N=400 (exceeds the 83ms/12fps budget). Render is negligible (~1ms at 400). Replace the brute-force scan with the 1D horizontal bucket grid the crowd research already specced (docs/research/crowd-awareness-simulation.md section 2.5) so the cap can rise well past ~400.

APPROACH: motion is surface-bound and ~1D-per-surface, so a 2D grid is unnecessary. In World.update, bucket characters by floor(pos.x / CELL) (cell sized to the largest interaction radius, e.g. CROWD_COH_RADIUS), built once per frame; hand each character only its own bucket +/- 1 as candidate neighbors instead of the full list. Keep the existing same-band (HSEP_Y_BAND) filtering inside steering. The Neighbor record shape and all steering/contagion rules stay the same — only the candidate set shrinks. Preserve correctness: a character must still see every neighbor within its interaction radius (cell >= max radius guarantees this with +/-1 lookup).

VERIFY: add a test asserting grid-based neighbor results match brute-force for a random layout (same set within radius). Re-run the benchmark approach from the capacity analysis and report new ms/frame at N=200/400/800 in the completion note. Full suite green. Add CELL/grid settings to settings.py. Touch only the listed files; do not change the Neighbor record or steering math.
