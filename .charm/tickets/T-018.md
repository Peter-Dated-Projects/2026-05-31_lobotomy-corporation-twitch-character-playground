BUG: World.spawn hardcodes a single character type so every viewer renders identically (and currently all hit the same fallback). The 10-character variety + per-viewer assignment never reaches the screen.

ROOT CAUSE: twitch_playground/sim/world.py line ~21 has `_CHARACTER_TYPE = 'default'  # one placeholder type for v0`, and spawn() (line ~91) calls `self.provider.get_sprite_set(_CHARACTER_TYPE)`. The provider's assign_character(username) already maps each distinct username to a stable CharacterDef, so passing the username through is all that's needed. dev.py already injects distinct usernames (viewer1, viewer2, ...).

FIX:
1. sim/world.py: pass the character's username (not the constant) to get_sprite_set in spawn(). Remove the now-dead `_CHARACTER_TYPE` constant. This completes the documented design (the constant's own comment says 'for v0'; the character-def convention says assign_character exists 'so each viewer gets a stable, distinct look'). Change nothing else.
2. tests/test_world.py: add a test that spawning several distinct usernames requests their own character_id from the provider (e.g. use a fake/recording AssetProvider that records the character_id passed to get_sprite_set, assert it equals each username). Keep using the existing test fixtures/placeholder provider style — do NOT require real ~/Downloads assets.

VERIFY: `uv run --project . pytest tests/test_world.py` green; full suite stays green. Match surrounding style. Do not touch the renderer, provider.py, or character_defs.py.
