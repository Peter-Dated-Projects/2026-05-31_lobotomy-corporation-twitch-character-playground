## Goal

Research and propose the best approach for swear-word filtering in the context of a Twitch chat-to-TTS pipeline. The speaker feature lets a viewer spend channel points and have their message read aloud by a robot character via TTS. We need to prevent profanity from being spoken.

## Constraints
- Must run on CPU (not GPU) -- the host machine may be a streaming PC with no spare GPU budget
- Must be fast enough to not add noticeable latency to the TTS pipeline (sub-50ms is ideal)
- Python-native preferred (the project is a Python app)
- The filter must catch the most common Twitch chat swear patterns: leetspeak substitutions (h3ll, f*ck, etc.), spaced-out letters (f u c k), common misspellings used to evade filters

## Scope of research
1. Text-level filtering (before sending to TTS) -- this is the primary recommendation since it is CPU-only
   - Evaluate: `better-profanity`, `profanity-filter`, `alt-profanity-check`, manual curated wordlist
   - Compare: evasion resistance (leetspeak, spacing), false-positive rate, install size, latency
2. Post-TTS audio-level filtering -- only if there is a lightweight CPU option worth noting
   - Are there any CPU-only audio classifiers for profanity? (e.g., Whisper-tiny transcribe-then-filter loop)
   - Is this worth the added complexity vs. text-level?
3. Recommendation: which approach (or combination) to use in this project

## Output
Write findings to `.charm/kb/swear-filter-research.md`. Include:
- Recommended approach with install command
- Code snippet showing how to integrate it in a Python string -> cleaned string function
- A note on what evasion vectors it does NOT cover and how to handle them
- Brief note on post-TTS audio filtering viability (1 paragraph max)

## Project context
The pipeline is approximately: viewer message (string) -> filter -> TTS synthesis -> audio playback. The filter step happens before TTS, so text-level is the right primary layer. The project uses `uv` for deps (pyproject.toml at repo root).
